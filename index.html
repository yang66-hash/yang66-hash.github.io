MediatorPattern

Mediator Pattern

在面向对象应用程序中，对象很少时，对象之间直接交互是可以的，但是若在程序中有很多的对象，那么直接交互就会导致混乱。

如果要增加一个新的类，则所有类的接口均需要改变。

一、这种点对点的交流存在的缺点：

  • 高耦合，复杂的方法调用；
  • 使得应用的可维护性降低；
  • 高耦合使得可重用性降低；

怎么解决这一问题呢？

重新设计交互结构，使用星型拓扑。

二、中介者模式类图

为什么中介者类必须拥有注册方法？

①使得中介者类知道所有Colleagues子类；

②注册方法可用于让所有的中介者保存所有参与对象。

为什么每个Colleague子类都必须保持中介者类的引用？

每个参与类必须保留中介类的引用，以便它可以调用中介类中的方法来执行某些操作。

三、中介者模式的用途

Mediator类负责所有的对象交互——①维持每个参与者类的引用；②提供调用参与对象的方法。

中介者模式的典型交互

任何两个不同的对象之间的交互都是通过Mediator实现的。①所有的对象把他们的信息发送给Mediator（调用Mediator的方法）；

Mediator发送信息到正确的对象上去实现应用请求；

中介者模式的优点：

①参与者类复用性变得更好；

②参与者对象之间的关系可以由中介者子类对象调节；

③有利于参与者类的单元测试，因为各个子类之间不存在相互之间的调用；

④有利于参与者类的修改，低耦合使得参与者类的修改不会影响到其他类

中介者模式的缺点:

使用中介者模式可能影响性能。

四、在Mediator类中应该写些什么？

①声明参与者类的私有成员变量

②写上register（Colleague colleague）方法， 注册对象类型应分别包含在注册方法的参数中。

③在每个注册方法中，将从参数传递的对象分配给相应的私有成员。

这样，参与对象就被”拉“进了中介者Mediator对象中。

VisitorPattern

Visitor Pattern

访问者模式的组件

访问者类为每一个Element子类准备一个访问者方法，方法名便表明向访问者发出观察请求类是哪一个。

访问者决定要访问哪个类。

访问者模式构件的解释

  • 具体Visitor实现子类必须实现Visitor类的接口中所声明的所有操作；
  • 提供算法环境、存储局部状态、积累遍历结构体的结果；
  • Element接口中必须要定义以visitor为参数的accept（）方法；
  • 具体的Element类必须实现被定义accept（）方法，也可实现一些其他方法；
  • ObjectStructure类中可以使用聚合结构，如ArrayList或者Vector将被访问者对象集合在一起，然后再使用一个循环语句批量调用accept（）方法。

什么时候使用访问者模式？

①当对象结构包含很多子类，而这些类有不同的接口你需要针对不同的类进行不同操作。

②许多不同的且不相干的操作需要施加于对象结构，而你想要避免这些不同的操作污染这些类；

③对象结构很少改变（例如税收的种类很少改变），但是你需要经常在这些结构体上增加新的运算（税率可能改变）。

访问者模式的优点

①访问者模式使得添加新方法变得更加轻松。相反，如果将功能扩展到多个类，则必须更改每个类以定义新操作。

②访问者聚集相关操作并分离无关操作；

访问者模式的缺点

添加一个新的具体待访问元素（concreteElement）是困难的。因为添加一个待访问元素，就需要在Visitor抽象类中添加相应的访问方法，每个具体实现类都需要实现这个方法。

什么是访问者模式的重点？

用于使得被访问者与访问者之间建立自动访问的accept方法是重点。该方法建立了两个层次类之间的关联。该方法使得自动访问成为可能。

注: ObjectStructure类对于批量访问很重要。另外使用此类，可以使得客户类包含较少的条件语句。能够有效地做到责任分离。

结论: 要写好accept方法与ObjectStructure

StatePattern

State Pattern

一、状态模式的组成部分

Context：

①定义客户程序所需要的接口；

②保持当前状态子类的对象；

③可以包含部分业务逻辑；

State：

定义一个接口封装与Context特定状态相关的行为。

State subclasses：

每一个子类实现一种与Context中状态相关的行为。

二、为什么使用状态模式？

①当对象的行为依赖于状态，而该对象必须根据其状态（在运行时)改变其行为 ；

②当操作带有大量状态相关的、多部分的条件语句。将每个条件分支封装在一个分离的类里面，使得软件工程师可以将状态当作对象对待，从而可以独立改变一种状态的代码；

三、关于状态的转换的建议

  • 客户类Client不负责状态转换
  • 客户类只创建一个“种子对象”，然后，传递给Context对象；
  • 再由Context对象根据状态变化的情况，负责创建所有的其它的对象
  • 或由State层次类决定状态的转换

StrategyBehavioral

Strategy Behavioral

一、什么时候使用策略模式？

当你想要处理一些仅仅行为上不同的相关的类的时候；

当你要隐藏实现细节；

当你要取消许多条件语句的时候。

二、Context的实现方式

当某个算法被调用的时候，Context类将Strategy类所需要的所有数据将一次性传递完毕；

Context将自己作为一个对象传递给Strategy类，然后Strategy类再利用该对象反过来调用Context类，以便获得某些数据与功能

三、策略模式的交互情况

①客户程序创建具体策略对象，并将此对象传递给Context；

②一个context的提出需要从客户程序一直到其策略类；

③客户程序与context交互。

四、策略模式的优点

①易扩展。将方法封装在单独的策略类中，可以独立于上下文更改方法，使其更易于理解、切换和扩展；

②若是策略层次类中需要添加新的子类，已存在的策略子类还有context类不会受影响。

五、策略模式的缺点

客户程序必须要知道不同类型的策略子类。

BridgePattern

Bridge Pattern

一、桥接模式的组件

  • 抽象部分，定义抽象爱你国际卡；
  • 重定义抽象，扩展抽象接口；
  • Implementor，定义实现类的接口，这个接口不必与抽象接口相同，事实上他们两者可以完全不同

抽象部分与实现部分分离开来，抽象部分提供高层功能，实现部分实现具体功能。

二、抽象部分与显示部分的责任

  • 实现接口仅提供原始操作（例如，写入数据库，写入数据文件等等）；
  • 抽象接口定义基于原始操作的高层操作（高层业务逻辑）；

三、桥接模式的优点

  • 将接口与实现解耦。体现在实现部分动态配置，取消编译时对实现部分的依赖，改变实现类不需要重新编译抽象类与客户类。
  • 改善了可扩展性。可以实现抽象部分和实现层次类的独立扩展；
  • 对用户实现了隐藏细节。

四、什么时候使用桥接模式？

当你想要避免抽象类及其实现类之间的永久绑定时，使用桥接模式。

AdapterPatternAndSomethingElse

Adapter Pattern

一、类适配器模式（Class adapter pattern）和对象适配器模式（Object adapter pattern）

在Java中，对象适配器是无法实现适配多个类的情况的，而使用对象适配器则可以适配多个类。

一个Target接口是可以有多个适配器的。

二、什么情况适合使用（类、对象）适配器？

①当你想使用一个已经存在的类，但是该类的接口并不是你想要的时候。

②当你想要创建一个可以复用的类的，该类与一些互不相关的接口不相容的类进行合作，或者说是你要改变许多子类的接口。在这种情况下，可以使用对象适配器模式。

三、适配器模式的应用主要体现在两个方面

  • 改变接口（主要的。添加功能也可以看做是改变接口，因为增加功能也就改变了接口）；
  • 增加功能；

四、两种适配器的不同点

在类适配器中，被适配类中所有的属性和方法都是被继承来的。

在对象适配器中，被适配类中往往只是选择一个或者是几个方法放入适配器类中

Lect3_MainProgram_subroutine-OOSystem

主程序-子程序系统以及面向对象系统

一、面向对象设计

面向对象设计的原理：基于信息隐藏

①什么叫做对象？包含状态（如，包含数据）以及语义（如，改变数据的规则）的事物。
②面相对象的基本概念：对象被用于统一算法抽象和数据抽象的思想。

③面向对象设计的性质：封装、继承、动态绑定以及复用和维护。

④面向对象设计的优点：

a) 程序容易维护(Easier maintenance)。由于对象对其客户程序隐藏其表示，因此可以在不影响客户程序的情况下改变实现。对象可以理解为独立实体；

b) 对象是可复用的组件(Reusable)；

c) 在一些系统中，对象是现实现实世界的映射；

d) 容易分解系统(Easy decomposition of a system) 对象将一组可访问的方法与其操作的数据捆绑在一起，允许设计人员将问题分解为互相交互的对象的集合。

MVCArchitecture

Theory of the MVC design pattern

一、MVC架构：在软件工程中，MVC架构是实现将用户与应用程序间的交互划分为三类组成部分的架构模式。

  • Model（业务逻辑）；
  • View（用户界面）；
  • Controller（用户输入）；
三者之间的连接部分：显式调用、隐式调用以及其他的机制（如HTTP协议）

二、Model的任务

①从数据库中提取数据，将数据存储到数据库中；

②负责业务逻辑的实现；

③负责数据验证，然后将数据存入到数据库中；

三、View的任务

①获取用户输入；

②向controller分发处理请求；

③按照Controller的知识，显式输出；

四、Controller的任务

①接受来自用户的请求；

②调用Model业务逻辑方法；

③调用View现实执行结果；

五、MVC架构的两个主要分离

①将View从Model中分离出来；

②将Controller从View中分离出来

六、MVC的优点

①容易增加或者改变视图

②容易独立地更新每个独立的软件模块

③代码易开发易维护

④业务逻辑更易测试

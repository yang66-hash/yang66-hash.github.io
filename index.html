一、翻译程序

将某种语言描述的程序翻译成等价的另一种语言描述的程序。

翻译程序分为解释程序和编译程序

什么叫解释程序？边解释边执行，不断读取源程序语句，解释语句，读取此语句需要的数据

什么叫编译程序？将源程序完整地转换成机器语言程序或汇编语言程序，然后再处理、执行的翻译程序

编译系统=编译程序+运行系统

中间代码生成中中间代码的特点：①简单规范；②与机器无关；③易于优化和转化；

编译程序的翻译分成词法分析、语法分析、语义分析与中间代码生成、代码优化、目标代码生成五个阶段

编译程序设计目标：规模小、速度快、诊断力强、可靠性高、可移植性好、可扩充性好；目标程序也要规模小、执行速度快；

ObseverPattern

Observer Pattern

观察者模式有助于在一组相关对象（Observers）和它们所依赖的对象（Subjects）之间设计一致的通信模型。

当你可以将一个程序中的对象分为主题与观察者的时候，可以使用如下的观察者模式进行设计。

一、有关于观察者模式

①一个主题可以有多个观察者，观察者需要知道主题的状态何时发生变化；

②主题保持一个动态列表，记录已经注册的观察者。

观察者模式必须要包含的活动

①观察者必须要注册自身；

②当主题状态改变之后，其必须通知所有被注册的观察者。

③一个观察者可以查询被观察者的状态。

二、通知观察者的两种策略

①将观察者所需要的数据全部推给观察者；（Push Model）

②主题提供接口，供观察者访问；（Pull Model）

三、使用观察者的好处

①可以动态添加观察者而不需要影响主题。

②当主题的状态或者逻辑发生变化的时候，观察者不受影响。

四、什么时候使用观察者？

①一个对象的状态改变必须反映在另一个对象中，而你又不想让两个对象保持高耦合

②你设计的框架将来需要添加许多观察者，而你希望以最小限度地改变程序

MediatorPattern

Mediator Pattern

在面向对象应用程序中，对象很少时，对象之间直接交互是可以的，但是若在程序中有很多的对象，那么直接交互就会导致混乱。

如果要增加一个新的类，则所有类的接口均需要改变。

一、这种点对点的交流存在的缺点：

  • 高耦合，复杂的方法调用；
  • 使得应用的可维护性降低；
  • 高耦合使得可重用性降低；

怎么解决这一问题呢？

重新设计交互结构，使用星型拓扑。

二、中介者模式类图

为什么中介者类必须拥有注册方法？

①使得中介者类知道所有Colleagues子类；

②注册方法可用于让所有的中介者保存所有参与对象。

为什么每个Colleague子类都必须保持中介者类的引用？

每个参与类必须保留中介类的引用，以便它可以调用中介类中的方法来执行某些操作。

三、中介者模式的用途

Mediator类负责所有的对象交互——①维持每个参与者类的引用；②提供调用参与对象的方法。

中介者模式的典型交互

任何两个不同的对象之间的交互都是通过Mediator实现的。①所有的对象把他们的信息发送给Mediator（调用Mediator的方法）；

Mediator发送信息到正确的对象上去实现应用请求；

中介者模式的优点：

①参与者类复用性变得更好；

②参与者对象之间的关系可以由中介者子类对象调节；

③有利于参与者类的单元测试，因为各个子类之间不存在相互之间的调用；

④有利于参与者类的修改，低耦合使得参与者类的修改不会影响到其他类

中介者模式的缺点:

使用中介者模式可能影响性能。

四、在Mediator类中应该写些什么？

①声明参与者类的私有成员变量

②写上register（Colleague colleague）方法， 注册对象类型应分别包含在注册方法的参数中。

③在每个注册方法中，将从参数传递的对象分配给相应的私有成员。

这样，参与对象就被”拉“进了中介者Mediator对象中。

VisitorPattern

Visitor Pattern

访问者模式的组件

访问者类为每一个Element子类准备一个访问者方法，方法名便表明向访问者发出观察请求类是哪一个。

访问者决定要访问哪个类。

访问者模式构件的解释

  • 具体Visitor实现子类必须实现Visitor类的接口中所声明的所有操作；
  • 提供算法环境、存储局部状态、积累遍历结构体的结果；
  • Element接口中必须要定义以visitor为参数的accept（）方法；
  • 具体的Element类必须实现被定义accept（）方法，也可实现一些其他方法；
  • ObjectStructure类中可以使用聚合结构，如ArrayList或者Vector将被访问者对象集合在一起，然后再使用一个循环语句批量调用accept（）方法。

什么时候使用访问者模式？

①当对象结构包含很多子类，而这些类有不同的接口你需要针对不同的类进行不同操作。

②许多不同的且不相干的操作需要施加于对象结构，而你想要避免这些不同的操作污染这些类；

③对象结构很少改变（例如税收的种类很少改变），但是你需要经常在这些结构体上增加新的运算（税率可能改变）。

访问者模式的优点

①访问者模式使得添加新方法变得更加轻松。相反，如果将功能扩展到多个类，则必须更改每个类以定义新操作。

②访问者聚集相关操作并分离无关操作；

访问者模式的缺点

添加一个新的具体待访问元素（concreteElement）是困难的。因为添加一个待访问元素，就需要在Visitor抽象类中添加相应的访问方法，每个具体实现类都需要实现这个方法。

什么是访问者模式的重点？

用于使得被访问者与访问者之间建立自动访问的accept方法是重点。该方法建立了两个层次类之间的关联。该方法使得自动访问成为可能。

注: ObjectStructure类对于批量访问很重要。另外使用此类，可以使得客户类包含较少的条件语句。能够有效地做到责任分离。

结论: 要写好accept方法与ObjectStructure

StatePattern

State Pattern

一、状态模式的组成部分

Context：

①定义客户程序所需要的接口；

②保持当前状态子类的对象；

③可以包含部分业务逻辑；

State：

定义一个接口封装与Context特定状态相关的行为。

State subclasses：

每一个子类实现一种与Context中状态相关的行为。

二、为什么使用状态模式？

①当对象的行为依赖于状态，而该对象必须根据其状态（在运行时)改变其行为 ；

②当操作带有大量状态相关的、多部分的条件语句。将每个条件分支封装在一个分离的类里面，使得软件工程师可以将状态当作对象对待，从而可以独立改变一种状态的代码；

三、关于状态的转换的建议

  • 客户类Client不负责状态转换
  • 客户类只创建一个“种子对象”，然后，传递给Context对象；
  • 再由Context对象根据状态变化的情况，负责创建所有的其它的对象
  • 或由State层次类决定状态的转换

StrategyBehavioral

Strategy Behavioral

一、什么时候使用策略模式？

当你想要处理一些仅仅行为上不同的相关的类的时候；

当你要隐藏实现细节；

当你要取消许多条件语句的时候。

二、Context的实现方式

当某个算法被调用的时候，Context类将Strategy类所需要的所有数据将一次性传递完毕；

Context将自己作为一个对象传递给Strategy类，然后Strategy类再利用该对象反过来调用Context类，以便获得某些数据与功能

三、策略模式的交互情况

①客户程序创建具体策略对象，并将此对象传递给Context；

②一个context的提出需要从客户程序一直到其策略类；

③客户程序与context交互。

四、策略模式的优点

①易扩展。将方法封装在单独的策略类中，可以独立于上下文更改方法，使其更易于理解、切换和扩展；

②若是策略层次类中需要添加新的子类，已存在的策略子类还有context类不会受影响。

五、策略模式的缺点

客户程序必须要知道不同类型的策略子类。

BridgePattern

Bridge Pattern

一、桥接模式的组件

  • 抽象部分，定义抽象爱你国际卡；
  • 重定义抽象，扩展抽象接口；
  • Implementor，定义实现类的接口，这个接口不必与抽象接口相同，事实上他们两者可以完全不同

抽象部分与实现部分分离开来，抽象部分提供高层功能，实现部分实现具体功能。

二、抽象部分与显示部分的责任

  • 实现接口仅提供原始操作（例如，写入数据库，写入数据文件等等）；
  • 抽象接口定义基于原始操作的高层操作（高层业务逻辑）；

三、桥接模式的优点

  • 将接口与实现解耦。体现在实现部分动态配置，取消编译时对实现部分的依赖，改变实现类不需要重新编译抽象类与客户类。
  • 改善了可扩展性。可以实现抽象部分和实现层次类的独立扩展；
  • 对用户实现了隐藏细节。

四、什么时候使用桥接模式？

当你想要避免抽象类及其实现类之间的永久绑定时，使用桥接模式。

Data flow Systems And Batch Sequential Architecture

数据流系统与批处理架构

流数据系统

一、流数据系统中数据的可用性控制着计算，设计的结构由进程之间的有序运动决定的、数据的模式是明确的系统。

一般来说，数据可以一任意的形式流动，数据到达组件激活，离开组件休眠。但是我们对几乎是线性的数据流系统以及相对简单，高度受限的循环结构更感兴趣。

关于控制流

我们关注程序中的控制流轨迹。数据可能伴随控制，但是数据不是主导，我们关系程序的执行顺序。

二、数据流系统在设计方面的注意事项

主要关注数据是怎样流动的。随着数据的流动，控制被激活。设计要考虑的注意事项是数据的可用性、数据变换和数据延迟。

数据流系统的开发方法论

将系统分解为一些模块，再将模块组合成图，再运行该图。

批处理系统

一、批处理系统概念

①批处理顺序过程由一系列处理步骤组成，中间是某种存储，如磁带、硬盘等。

②每个步骤都将对输入存储器执行一些操作，以获取一些有用的信息或修改源存储器的内容，然后将结果数据保存到其接收器存储器。

二、批处理系统性质

①处理步骤是独立的程序，任何两个处理步骤之间不存在交互

②每一步在彻底运行完成之后，下一步才能开始

③数据作为一个整体在各个步骤之间传输（理解为一种特殊的数据流；数据以块状的形态流动）；

MainProgram_subroutine-OOSystem

主程序-子程序系统

一、非结构化编程的缺点：
  • 很难追踪程序逻辑；
  • 很难合并其它代码；
  • 修改代码并不容易(都是goto惹的祸)；
  • 很难测试特定部分的代码；
二、结构化编程

结构化编程是一种编程范式，旨在通过广泛使用：

子程序(subroutines)、块状结构 (block s tructures)和循环结构：for和while循环来提高计算机程序的清晰度、质量和开发时间。

结构化程序定理：只要一种编程语言可以依三个方式（序列结构、选择结构和迭代结构）组合其子程序及调整控制流程，则每个可计算函数都可以用此种编程语言来表示（实现任何算法）。这是结构化编程的理论基础。

三、调用——返回体系机构
  • 调用返回架构使用分割征服策略；
  • 将整个系统分为更小的子系统，以减少复杂度；
  • 调用返回的执行顺序被一个单线程控制；

面向对象设计

面向对象设计的原理：基于信息隐藏

①什么叫做对象？
包含状态（如，包含数据）以及语义（如，改变数据的规则）的事物。
②面相对象的基本概念：
对象被用于统一算法抽象和数据抽象的思想。

③面向对象设计的性质：

封装、继承、动态绑定以及复用和维护。

④面向对象设计的优点：

a) 程序容易维护(Easier maintenance)。由于对象对其客户程序隐藏其表示，因此可以在不影响客户程序的情况下改变实现。对象可以理解为独立实体；

b) 对象是可复用的组件(Reusable)；

c) 在一些系统中，对象是现实现实世界的映射；

d) 容易分解系统(Easy decomposition of a system) 对象将一组可访问的方法与其操作的数据捆绑在一起，允许设计人员将问题分解为互相交互的对象的集合。

结构化设计与面向对象设计的区别

  • 结构化设计结果是一颗树，其每个节点都是函数；这些函数按照一定的顺序调用，以便取得预期功能。
  • 面向对象设计得到类图，由一群类组成；每个类都将相关的数据与施加于数据上的操作组织在一起。

Dataflow Systems Pipes and Filters Architecture

管道-过滤器架构是由处理数据的过滤器以及在过滤器之间传输数据的管道组成

这种架构适用于流数据的处理。

过滤器的功能：递增地将某些源数据转换为其他数据
  • 添加信息。计算和添加信息使数据得以丰富。
  • 变换数据。通过改变状态来传输数据。
  • 流对流变换。流到流的转换。
  • 不保留状态。在过滤器的两次实例化之间不保留任何状态，即过滤器并不记录每一次发生了什么操作。
管道的功能：把数据从一个过滤器的输出移到另一个的输入。

架构的整体运行机制：运行管道与过滤器直到没有任何计算，数据的搬运只是进行数据处理的媒介。

书记在以数据流的方式从左端流入，流经各个过滤器之后，对数据进行处理，但与处理完的数据，立即写到下游管道。

批处理架构和管道-过滤器架构的比较

两者的相似之处：

处理过程之间互不调用

不同点：

  • 数据处理方式不同。前者数据是一块状形式传输的，而后者是以流的形式；
  • 管道-过滤器架构。所处理的数据量也有所不同。前者数据量是有限的，后者是无限的；

管道-过滤器架构的优缺点：

  • 复用性好：每个过滤器的功能都是独立的；
  • 容易扩展：在处理过程中加入过滤器，能轻易实现新功能的扩展；
  • 容易修改：因为过滤器在逻辑上是独立的，容易实现系统的修改；

管道-过滤器架构也存在缺点：

  • 人机交互性差：纯粹的管道-过滤器架构不存在人机交互，但是我们实际应用时，可以给每个过滤器加上用户图像界面；
  • 浪费内存：因为数据需要不断的从某个过滤器的输入端口复制到另一个的输出端口，因此内存空间利用率比较低；

Event-based Software Architecture

基于事件系统

基于事件系统是这样一种系统，在这个系统中，过程并不会直接被调用（间接或者是隐式调用）

组件的结构：隐式调用风格中的组件是一个模块，其接口同时提供

  • 一个过程集合（C++中的函数、java中的方法）;
  • 一个事件的集合；

事件处理策略：当一个事件被广播了，系统将自动调用那些已经注册了的组件，那么怎样将事件发送到已经注册了的组件中呢？

  • 策略一：有独立事件调度模块的系统；

  • 策略二：无中心事件调度模块的系统(这个模型也常被叫做observable/Observer)；

观察者模式可以被用来设计与实现比较简单的事件系统

事件调度模块的责任是什么呢？

  • 接受事件；
  • 分发事件；

分发的方式：

①广播事件给系统的全部模块；

②仅发送事件给那些已注册了该事件的模块；

Client-server Architecture

客户端(Client)：服务请求者，软件；

服务器(Server)：服务提供者，软件；

交互方式：客户端与服务器互相交互；

硬件无关：此架构与硬件无关；

平台无关：此架构与平台( Windows, Unix）无关；

客户端服务器架构的动机：

  • 在多个用户之间共享资源；
  • 客户端软件与最终用户交互；
  • 服务器软件与共享资源交互；

一、客户端/服务器架构的演化

①一层客户端/服务器（……-1970’s）：

大型机计算机应用程序可视为一层客户端/服务器应用程序

文件共享架构：

PC从文件服务器请求文件；业务逻辑全部在客户端，共享资源在服务器上；在数据量较低的时候，工作情况尚可；

②两层客户端-服务器架构：

数据库服务器取代文件服务器；

2层客户端/服务器的内容：

第1层：GUI+业务逻辑；

第2层：数据库+共享资源+业务逻辑(少量的程序)；

两层结构很简单

用于小型部门决策支持；

用于支持简单的Web发布应用程序；

两层客户端-服务器架构的优点：

与文件共享系统相比，因为两层客户端-服务器发送请求与回答而不是文件，从而降低了网络使用。

两层客户端-服务器架构的缺点：
  • 局域网应用；
  • 受限于供应商；
  • 数据库开发商的锁定；
  • 程序配置困难；
  • 程序维护困难；
  • 软件升级困难；
③3层客户端-服务器架构

第1层：客户端 GUI；

第2层：应用服务器Business Logic

第3层：数据库服务器 Data；

在两层的客户端-服务器架构的客户端（UI）与数据库服务器(database server)之间增加了一个中间层，叫做应用服务器(application server)；

两层客户端和三层客户端的比较
  • 三层客户端-服务器架构更容易维护；
  • 三层客户端-服务器架构可以有瘦客户端；
  • 在三客户端-服务器架构中，各层开发语言可以是互相独立的；
  • 三层客户端-服务器架构可以连接到多个数据库；
3层层次架构与3层客户端-服务器架构的比较

三层层次架构（包括显示层，应用层与永久数据保持层）中的层指的是逻辑意义上的层。这些层，可以部署在同一台计算机上，当然也可以部署在不同的计算机上。

三层客户端-服务器架构里面所涉及的层指的是物理意义上的层。

  • 客户端运行在PC上；
  • 应用层被部署与运行在应用服务器上；
  • 数据层包含数据库与数据库管理系统软件，被部署与运行在数据库服务器上。

AdapterPattern

Adapter Pattern

一、类适配器模式（Class adapter pattern）和对象适配器模式（Object adapter pattern）

在Java中，对象适配器是无法实现适配多个类的情况的，而使用对象适配器则可以适配多个类。

一个Target接口是可以有多个适配器的。

二、什么情况适合使用（类、对象）适配器？

①当你想使用一个已经存在的类，但是该类的接口并不是你想要的时候。

②当你想要创建一个可以复用的类的，该类与一些互不相关的接口不相容的类进行合作，或者说是你要改变许多子类的接口。在这种情况下，可以使用对象适配器模式。

三、适配器模式的应用主要体现在两个方面

  • 改变接口（主要的。添加功能也可以看做是改变接口，因为增加功能也就改变了接口）；
  • 增加功能；

四、两种适配器的不同点

在类适配器中，被适配类中所有的属性和方法都是被继承来的。

在对象适配器中，被适配类中往往只是选择一个或者是几个方法放入适配器类中

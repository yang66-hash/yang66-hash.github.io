数据流系统与批处理架构

流数据系统

一、流数据系统中数据的可用性控制着计算，设计的结构由进程之间的有序运动决定的、数据的模式是明确的系统。

一般来说，数据可以一任意的形式流动，数据到达组件激活，离开组件休眠。但是我们对几乎是线性的数据流系统以及相对简单，高度受限的循环结构更感兴趣。

关于控制流

我们关注程序中的控制流轨迹。数据可能伴随控制，但是数据不是主导，我们关系程序的执行顺序。

二、数据流系统在设计方面的注意事项

主要关注数据是怎样流动的。随着数据的流动，控制被激活。设计要考虑的注意事项是数据的可用性、数据变换和数据延迟。

数据流系统的开发方法论

将系统分解为一些模块，再将模块组合成图，再运行该图。

批处理系统

一、批处理系统概念

①批处理顺序过程由一系列处理步骤组成，中间是某种存储，如磁带、硬盘等。

②每个步骤都将对输入存储器执行一些操作，以获取一些有用的信息或修改源存储器的内容，然后将结果数据保存到其接收器存储器。

二、批处理系统性质

①处理步骤是独立的程序，任何两个处理步骤之间不存在交互

②每一步在彻底运行完成之后，下一步才能开始

③数据作为一个整体在各个步骤之间传输（理解为一种特殊的数据流；数据以块状的形态流动）；

主程序-子程序系统

一、非结构化编程的缺点：
  • 很难追踪程序逻辑；
  • 很难合并其它代码；
  • 修改代码并不容易(都是goto惹的祸)；
  • 很难测试特定部分的代码；
二、结构化编程

结构化编程是一种编程范式，旨在通过广泛使用：

子程序(subroutines)、块状结构 (block s tructures)和循环结构：for和while循环来提高计算机程序的清晰度、质量和开发时间。

结构化程序定理：只要一种编程语言可以依三个方式（序列结构、选择结构和迭代结构）组合其子程序及调整控制流程，则每个可计算函数都可以用此种编程语言来表示（实现任何算法）。这是结构化编程的理论基础。

三、调用——返回体系机构
  • 调用返回架构使用分割征服策略；
  • 将整个系统分为更小的子系统，以减少复杂度；
  • 调用返回的执行顺序被一个单线程控制；

面向对象设计

面向对象设计的原理：基于信息隐藏

①什么叫做对象？
包含状态（如，包含数据）以及语义（如，改变数据的规则）的事物。
②面相对象的基本概念：
对象被用于统一算法抽象和数据抽象的思想。

③面向对象设计的性质：

封装、继承、动态绑定以及复用和维护。

④面向对象设计的优点：

a) 程序容易维护(Easier maintenance)。由于对象对其客户程序隐藏其表示，因此可以在不影响客户程序的情况下改变实现。对象可以理解为独立实体；

b) 对象是可复用的组件(Reusable)；

c) 在一些系统中，对象是现实现实世界的映射；

d) 容易分解系统(Easy decomposition of a system) 对象将一组可访问的方法与其操作的数据捆绑在一起，允许设计人员将问题分解为互相交互的对象的集合。

结构化设计与面向对象设计的区别

  • 结构化设计结果是一颗树，其每个节点都是函数；这些函数按照一定的顺序调用，以便取得预期功能。
  • 面向对象设计得到类图，由一群类组成；每个类都将相关的数据与施加于数据上的操作组织在一起。

Dataflow Systems Pipes and Filters Architecture

管道-过滤器架构是由处理数据的过滤器以及在过滤器之间传输数据的管道组成

这种架构适用于流数据的处理。

过滤器的功能：递增地将某些源数据转换为其他数据
  • 添加信息。计算和添加信息使数据得以丰富。
  • 变换数据。通过改变状态来传输数据。
  • 流对流变换。流到流的转换。
  • 不保留状态。在过滤器的两次实例化之间不保留任何状态，即过滤器并不记录每一次发生了什么操作。
管道的功能：把数据从一个过滤器的输出移到另一个的输入。

架构的整体运行机制：运行管道与过滤器直到没有任何计算，数据的搬运只是进行数据处理的媒介。

书记在以数据流的方式从左端流入，流经各个过滤器之后，对数据进行处理，但与处理完的数据，立即写到下游管道。

批处理架构和管道-过滤器架构的比较

两者的相似之处：

处理过程之间互不调用

不同点：

  • 数据处理方式不同。前者数据是一块状形式传输的，而后者是以流的形式；
  • 管道-过滤器架构。所处理的数据量也有所不同。前者数据量是有限的，后者是无限的；

管道-过滤器架构的优缺点：

  • 复用性好：每个过滤器的功能都是独立的；
  • 容易扩展：在处理过程中加入过滤器，能轻易实现新功能的扩展；
  • 容易修改：因为过滤器在逻辑上是独立的，容易实现系统的修改；

管道-过滤器架构也存在缺点：

  • 人机交互性差：纯粹的管道-过滤器架构不存在人机交互，但是我们实际应用时，可以给每个过滤器加上用户图像界面；
  • 浪费内存：因为数据需要不断的从某个过滤器的输入端口复制到另一个的输出端口，因此内存空间利用率比较低；

Event-based Software Architecture

基于事件系统

基于事件系统是这样一种系统，在这个系统中，过程并不会直接被调用（间接或者是隐式调用）

组件的结构：隐式调用风格中的组件是一个模块，其接口同时提供

  • 一个过程集合（C++中的函数、java中的方法）;
  • 一个事件的集合；

事件处理策略：当一个事件被广播了，系统将自动调用那些已经注册了的组件，那么怎样将事件发送到已经注册了的组件中呢？

  • 策略一：有独立事件调度模块的系统；

  • 策略二：无中心事件调度模块的系统(这个模型也常被叫做observable/Observer)；

观察者模式可以被用来设计与实现比较简单的事件系统

事件调度模块的责任是什么呢？

  • 接受事件；
  • 分发事件；

分发的方式：

①广播事件给系统的全部模块；

②仅发送事件给那些已注册了该事件的模块；

Client-server Architecture

客户端(Client)：服务请求者，软件；

服务器(Server)：服务提供者，软件；

交互方式：客户端与服务器互相交互；

硬件无关：此架构与硬件无关；

平台无关：此架构与平台( Windows, Unix）无关；

客户端服务器架构的动机：

  • 在多个用户之间共享资源；
  • 客户端软件与最终用户交互；
  • 服务器软件与共享资源交互；

一、客户端/服务器架构的演化

①一层客户端/服务器（……-1970’s）：

大型机计算机应用程序可视为一层客户端/服务器应用程序

文件共享架构：

PC从文件服务器请求文件；业务逻辑全部在客户端，共享资源在服务器上；在数据量较低的时候，工作情况尚可；

②两层客户端-服务器架构：

数据库服务器取代文件服务器；

2层客户端/服务器的内容：

第1层：GUI+业务逻辑；

第2层：数据库+共享资源+业务逻辑(少量的程序)；

两层结构很简单

用于小型部门决策支持；

用于支持简单的Web发布应用程序；

两层客户端-服务器架构的优点：

与文件共享系统相比，因为两层客户端-服务器发送请求与回答而不是文件，从而降低了网络使用。

两层客户端-服务器架构的缺点：
  • 局域网应用；
  • 受限于供应商；
  • 数据库开发商的锁定；
  • 程序配置困难；
  • 程序维护困难；
  • 软件升级困难；
③3层客户端-服务器架构

第1层：客户端 GUI；

第2层：应用服务器Business Logic

第3层：数据库服务器 Data；

在两层的客户端-服务器架构的客户端（UI）与数据库服务器(database server)之间增加了一个中间层，叫做应用服务器(application server)；

两层客户端和三层客户端的比较
  • 三层客户端-服务器架构更容易维护；
  • 三层客户端-服务器架构可以有瘦客户端；
  • 在三客户端-服务器架构中，各层开发语言可以是互相独立的；
  • 三层客户端-服务器架构可以连接到多个数据库；
3层层次架构与3层客户端-服务器架构的比较

三层层次架构（包括显示层，应用层与永久数据保持层）中的层指的是逻辑意义上的层。这些层，可以部署在同一台计算机上，当然也可以部署在不同的计算机上。

三层客户端-服务器架构里面所涉及的层指的是物理意义上的层。

  • 客户端运行在PC上；
  • 应用层被部署与运行在应用服务器上；
  • 数据层包含数据库与数据库管理系统软件，被部署与运行在数据库服务器上。

P2P architecture

P2P使得任何的网络感知设备可以为另外一个网络感知设备提供服务；P2P网络的一个节点既是客户端，又是服务器；P2P计算通过直接交换的方式共享资源；

P2P架构的优点

①便宜的设备协作、利用集体力量

②节点既是客户端又是服务器；

③无传统意义下的服务器超载现象；

纯P2P架构的性质
  • 节点既是客户端又是服务器
  • 没有中心服务器
  • 两种路由结构：①分布式目录； ②直接发送消息；
